## Git 워크플로우 규칙

1. **브랜치 전략**
    -. `main` 브랜치는 항상 배포 가능한 상태를 유지합니다.
    -. 기능 개발은 `feature/이슈번호-설명` 형식의 브랜치에서 진행합니다.
    -. 버그 수정은 `fix/이슈번호-설명` 형식의 브랜치에서 진행합니다.
    -. 배포 준비는 `release/버전` 브랜치에서 진행합니다.
    -. 긴급 수정은 `hotfix/이슈번호-설명` 브랜치에서 진행합니다.

2. **커밋 메시지 규칙**
    - 커밋 메시지는 명확하고 간결하게 작성합니다.
    - 커밋 메시지 형식: `[타입] 이슈번호: 작업 내용`
      - 예시: `[feat] #123: 로그인 기능 추가`
    - 타입 예시: feat, fix, docs, style, refactor, test, chore

3. **PR(Pull Request) 규칙**
    - PR은 반드시 리뷰어를 지정합니다.
    - PR 설명에 작업 내용을 상세히 작성합니다.
    - CI가 통과된 후에만 머지합니다.
    - 최소 1명 이상의 리뷰어 승인을 받아야 머지할 수 있습니다.

4. **코드 리뷰**
    - 코드 리뷰 시에는 코드 스타일, 고직, 보안, 성능 등을 중점적으로 확인합니다.
    - 리뷰어는 개선 사항이 있으면 코멘트를 남기고, 필요시 수정 요청을 합니다.

5. **기타**
    - 불필요한 파일이나 디버깅 코드는 커밋하지 않습니다.
    - 커밋 전 `git pull --rebase`로 최신 코드를 반영합니다.
